# 6주차 — 교착상태 (Deadlock)

## 🎯 학습 목표

- 교착상태를 이해한다.
  - 정의와 발생 원인
  - 발생 조건
  - 해결 방법

---

## 1. 교착상태(Deadlock)란?

**정의**: 2개 이상의 프로세스가 서로의 작업 완료를 기다리며 **더 이상 진행되지 않는 상태**.

```
 Process A ──wait──→ Resource 1 ←──held──  Process B
     │                                         │
     └──── held ────→ Resource 2 ←──wait───────┘
          무한 대기 순환 ↻
```

### ⚠️ 교착상태 vs 기아(아사)

| 구분 | 교착상태 (Deadlock) | 기아 (Starvation/아사) |
|------|---------------------|------------------------|
| **원인** | 자연적 발생 — 프로세스 간 자원 얽힘 | **스케줄러 정책 문제** |
| **해결** | 자원/프로세스 강제 종료 (재부팅) | Aging으로 해결 |
| **예시** | 식사하는 철학자 문제 | 낮은 우선순위 프로세스 대기 |

> 💡 **핵심 차이**: 교착상태는 **자연스럽게** 발생하는 현상, 기아는 **정책** 문제.

---

## 2. 자원 할당 그래프 (Resource Allocation Graph, RAG)

교착상태를 **시각적으로 분석**하는 도구.

### 구성

- **G = (V, E)**
- V = Process 집합(P) ∪ Resource 집합(R)
- E (Edge) = **요청 간선** (P → R) + **할당 간선** (R → P)

### 예시

```
   P1 ──request──→ R1 ──allocate──→ P2
    ↑                                │
    └─────── allocate ─── R2 ←── request
    
   사이클 존재? → 교착상태 가능성 ↑
```

### 다중 자원 (Multiple Resource)

동일 종류 자원이 여러 개일 때는 사각형 안에 **작은 동그라미로 개수 표시**.

---

## 3. 식사하는 철학자 문제 (Dining Philosophers Problem)

**모든 운영체제 교재의 단골 예제**.

```
        철학자 1
       /        \
      P4          P1
     🍴          🍴
    (4)          (1)
   P4             P1
     🍴          🍴
      P3          P2
       \        /
        철학자 3
```

**규칙**:
- 4명의 철학자, 4개의 포크
- **왼쪽 포크 → 오른쪽 포크** 순서로 잡아야 식사 가능

**문제 상황**:
- 모두가 동시에 왼쪽 포크를 잡으면?
- → 오른쪽 포크는 옆 사람이 잡고 있어 대기
- → **모두가 무한 대기** → **교착상태 발생!**

---

## 4. 교착상태의 4가지 필요조건

**모두 만족**해야 교착상태 발생. 하나라도 깨지면 발생 X.

| # | 조건 | 영어 | 의미 |
|---|------|------|------|
| 1 | **상호 배제** | Mutual Exclusion | 자원을 공유할 수 없음 (한 번에 한 프로세스) |
| 2 | **비선점** | Non-preemption | 자원을 강제로 뺏을 수 없음 |
| 3 | **점유와 대기** | Hold & Wait | 자원을 가진 채로 다른 자원을 기다림 |
| 4 | **원형 대기** | Circular Wait | 요청이 원형(환형, 순환) 형태 |

> 📌 **용어 주의**: 같은 "Circular Wait"를 **원형대기**, **환형대기**, **순환대기** 등 다르게 번역함 (정보처리기사 등 시험에서 주의).

### 식사하는 철학자 문제 적용

| 조건 | 철학자 문제 |
|------|-------------|
| 상호 배제 | 포크를 2명이 동시에 못 씀 |
| 비선점 | 포크를 뺏을 수 없음 |
| 점유와 대기 | 왼쪽 잡고 오른쪽 기다림 |
| 원형 대기 | 1→2→3→4→1 순환 |

---

## 5. 교착상태 해결 방법 4가지

| 방법 | 개념 | 현실성 |
|------|------|--------|
| **1. 예방** (Prevention) | 4조건 중 하나를 원천 차단 | ❌ 비현실적 |
| **2. 회피** (Avoidance) | 안전 상태에서만 자원 할당 | ⚠️ 오버헤드 |
| **3. 검출 & 회복** (Detection & Recovery) | 발생 후 탐지 → 복구 | ✅ 사용됨 |
| **4. 무시** (Ignore) | 문제 시 재부팅 | ✅ **대부분의 OS** |

---

## 6. 교착상태 예방 (Prevention) — ❌ 불가능

4가지 조건 중 하나를 깨뜨려 **원천 차단**하는 방법.

| 조건 | 예방 방법 | 문제점 |
|------|----------|--------|
| **상호 배제 예방** | 자원 공유 허용 | 많은 자원이 본래 배타적 (프린터 등) → **불가능** |
| **비선점 예방** | 자원 빼앗기 허용 | 일관성 훼손 (중간 데이터 손상) → **불가능** |
| **점유와 대기 예방** | 필요한 자원을 한 번에 모두 할당 | 자원 낭비 → **비효율** |
| **원형 대기 예방** | 자원 번호 순서대로만 할당 | 자원 낭비 → **비효율** |

→ **결론**: 실무에서는 사용 어려움.

## 7. 교착상태 회피 (Avoidance) — ⚠️ 오버헤드

**안전 상태(Safe State)** 내에서만 자원 할당. 불안정 상태로 갈 것 같으면 거부.

### 안전/불안정 상태

```
    자원의 총수
         │
         │      안전 상태 (삼각형)
         │        ╱
         │      ╱
         │    ╱
         │  ╱
         └──────────── 할당된 자원
```

OS가 계속 **모니터링**하며 안전 영역에서만 자원 할당.

### 🏦 은행원 알고리즘 (Banker's Algorithm)

다익스트라(Dijkstra)가 제안. 은행원이 대출 가능 여부를 판단하듯, 자원 할당이 **안전한지** 사전 검사.

**예시**:

| Process | Max | Allocation | Expect |
|---------|-----|-----------|--------|
| P1 | 5 | 2 | 3 |
| P2 | 6 | 4 | 2 |
| P3 | 10 | 6 | 4 |

**가용 자원 = 2개일 때**:
- P1 필요=3 → ❌ 못 줌
- **P2 필요=2 → ✅ 할당!** (P2 먼저 수행)
- P2 종료 → 6개 반납 → 가용=6개
- 이제 P1이든 P3든 할당 가능

**가용 자원 = 1개일 때**:
- 어느 프로세스도 만족 못 함 → **불안정 상태** → 교착 위험

### ❌ 단점

- 시스템을 **항상 모니터링** → **운영체제 오버헤드 ↑**
- 자원·프로세스 수가 고정되어야 함
- 사용률 낮은 자원 → 효율 ↓

## 8. 교착상태 검출 & 회복 (Detection & Recovery) ✅ 실무

### 교착 상태 검출 (Detection)

**운영체제가 주기적으로 모니터링** → 교착상태로 의심되면 검출.

> 🖼️ **한글 프로그램 예시**: 저장하려는데 창이 하얗게 돌아가기만 하면
> → "응답 없음" 감지 → "프로그램 닫기" 창 표시

### 검출 방법: 자원 할당 그래프의 사이클

```
단일 자원: 사이클 = 교착 상태
다중 자원: 사이클이 있어도 교착 아닐 수 있음 (더 정교한 검사 필요)
```

### 📊 데이터베이스의 특수 문제: 일관성

```
[100만 원 입금 루틴 실행 중...]
       ↓
   프로세스 강제 종료  ← 100만 원 유실! 💸
```

**해결**: **체크포인트(Checkpoint) / 스냅샷(Snapshot)**

```
체크포인트 저장 → 작업 수행 → 실패 시 롤백(Roll back)
```

### 교착 상태 회복 (Recovery)

**프로세스를 강제로 종료**하여 자원 회수.

| 방식 | 설명 |
|------|------|
| **모든 프로세스 종료** | 즉시 해결, 작업 모두 유실 |
| **순차적 종료** | 우선순위에 따라 하나씩 종료, 교착 해소까지 반복 |

---

## 9. 교착상태 무시 (Ignore) — **대부분 OS의 선택**

교착상태에 대해 **아무런 대책 없이** 방치.

- 시스템이 멈추면 사용자가 **재부팅**
- **타조(Ostrich) 알고리즘** — 타조가 위험 시 머리를 땅에 묻듯 무시
- 문제 발생 빈도가 낮으면 **모니터링 비용 > 복구 비용**이므로 합리적

> 🖥️ **Windows/macOS/대부분 Linux**가 이 방식. 컴퓨터 멈추면 껐다 켜는 이유.

## 10. 그래프 분석 연습

### 예제 #1: 단일 자원

```
R1, R2  (각 1개)
P1 → R1 (wait)
R1 → P2 (held)
P2 → R2 (wait)
R2 → P1 (held)

→ P1-R1-P2-R2-P1 사이클
→ 교착상태 발생! ✅
```

### 예제 #2: 다중 자원

```
R1 (2개), R2 (1개)
P1 → R1 (wait)
R1 → P2, P3 (held)
P2 → R2 (wait)
R2 → P3 (held)

→ 사이클은 있지만 P3이 자원 반납하면 풀림
→ 교착상태 아님 ❌
```

## 📝 핵심 요약

1. **교착상태 = 서로 자원 얽혀 무한 대기** (자연 발생)
2. **기아 ≠ 교착상태** (기아는 스케줄러 정책 문제)
3. 4가지 필요조건: **상호배제, 비선점, 점유와대기, 원형대기**
4. 해결 4방법: **예방(X) / 회피(은행원) / 검출&회복(롤백) / 무시(재부팅)**
5. **식사하는 철학자 문제**가 대표 예제
6. **대부분의 현대 OS는 "무시" 전략** 사용
