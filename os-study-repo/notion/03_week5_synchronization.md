# 5주차 — 프로세스 동기화 (IPC & 임계구역)

## 🎯 학습 목표

- 프로세스 간 통신(IPC)를 이해한다.
- 임계구역을 이해한다.

---

## 1. IPC의 배경

프로세스는 메모리에 **독립적으로** 만들어집니다. → 서로 데이터 공유 불가.

```
┌──────────┐    독립      ┌──────────┐
│ Process A│   (분리)     │ Process B│
│  Memory  │              │  Memory  │
└──────────┘              └──────────┘
     ❌ 직접 통신 불가
```

→ **IPC (Inter-Process Communication)** 기법이 필요!

> 💡 반면 **멀티스레드**는 같은 프로세스 내에서 메모리를 공유하므로 IPC가 필요 없습니다.

---

## 2. 통신 분류

### 방향성에 따른 분류

| 유형 | 설명 | 예 |
|------|------|-----|
| **단방향 (Simplex)** | 한 방향으로만 | 파이프 |
| **양방향 (Duplex)** | 양쪽 모두 가능 | 소켓, 공유 메모리 |
| **반양방향 (Half-duplex)** | 한 번에 한 방향 | 무전기 |

### 동시성에 따른 분류

| 유형 | 설명 | 방식 |
|------|------|------|
| **대기 있는 통신 (Blocking)** | 데이터 받을 때까지 대기 | Interrupt 기반 |
| **대기 없는 통신 (Non-Blocking)** | 대기하지 않고 다른 일 수행 | Polling 기반 |

---

## 3. IPC 기법 7가지 총정리

| 기법 | 방향성 | 동시성 | 사용 범위 | 특징 |
|------|--------|--------|----------|------|
| **시그널** | 단방향 | 비동기 | 동일 호스트 | 제어 목적, 데이터 전달 ❌ |
| **인터럽트** | 단방향 | 비동기 | 동일 호스트 | 이벤트 기반, 매우 빠름 |
| **공유 파일** | 양방향 | 동기 | 동일 호스트 | 디스크 I/O → 느림 |
| **파이프** | 단방향 | 동기 | 동일 호스트 | 부모-자식 간 주로 사용 |
| **메시지 큐** | 양방향 | 비동기 | 동일 호스트 | 큐 방식, 부모-자식 무관 |
| **공유 메모리** | 양방향 | 비동기 | 동일 호스트 | **가장 빠름**, 동기화 필요 |
| **소켓** | 양방향 | 동기/비동기 | **네트워크** | 원격 통신 가능 |

---

## 4. 각 IPC 기법 상세

### 🚦 시그널 (Signal)

한 프로세스가 다른 프로세스에게 **신호만** 전송 (데이터 X).

- 예: `SIGKILL` (9번) — 프로세스 강제 종료
- `SIGSEGV` (11번) — 세그멘테이션 폴트

### ⚡ 인터럽트 vs 폴링 (Polling)

**인터럽트**:
```
프로세스 실행 중 → 이벤트 발생 → ISR 실행 → PC 기준 복귀
```

**폴링**:
```
프로세스가 주기적으로 "자료 있어?" 확인 → CPU 낭비 가능
```

| 구분 | 인터럽트 | 폴링 |
|------|---------|------|
| **주체** | 하드웨어/OS | 프로세스 |
| **효율** | 높음 (이벤트 시에만) | 낮음 (계속 확인) |
| **응답성** | 빠름 | 느림 |

### 📁 공유 파일

하드디스크의 파일을 통해 데이터 공유. C언어에선 `open()` 함수 사용 → **파일 기술자(fd)** 반환.

```c
int fd = open("shared.txt", O_RDWR);
```

- **순차 파일**: fd는 한 방향으로만 진행 (되돌릴 수 없음)
- 하드디스크 I/O → **느림**

### 🔗 파이프 (Pipe)

커널 영역에 **일정한 공간(파이프)**을 만들어 부모 → 자식 단방향 통신.

```
┌─────────────────────────┐
│     Kernel Area         │
│  ┌───────────────────┐  │
│  │      Pipe         │  │
│  └───────────────────┘  │
│    ↑              ↓     │
│   Parent        Child   │
└─────────────────────────┘
```

#### 양방향 파이프?

- 하나의 파이프로 양방향 시도 → **데이터 충돌 발생**
- 해결: **단방향 파이프 2개**로 구성

### 📬 메시지 큐

큐(Queue) 방식으로 데이터 주고받음. `enqueue` / `dequeue`.

- 양방향 통신 가능
- **부모-자식 관계가 아니어도** 사용 가능

### 🧠 공유 메모리

메모리에 별도로 **할당된 공유 영역** 활용 → 가장 빠름.

```bash
# 현재 공유 메모리 상태 확인
$ ipcs
$ ipcrm  # 제거
```

⚠️ **동기화 필수** — Race Condition 방지를 위해 세마포어/뮤텍스와 함께 사용.

### 🌐 소켓 (Socket)

**네트워크**를 통한 외부 컴퓨터와의 통신.

```
프로세스 A (컴퓨터 1)              프로세스 B (컴퓨터 2)
    │                                     │
  ┌─────┐                              ┌─────┐
  │Socket│ ←──── TCP/IP 네트워크 ────→│Socket│
  └─────┘                              └─────┘
```

**🎁 택배 비유**:
1. 데이터를 소켓에 담음 (택배 박스 포장)
2. TCP/IP 주소로 발송
3. 수신 측에서 주소 확인 후 박스 개봉 → 데이터 확인

> 모든 프로그래밍 언어에서 **동일한 구조**로 구현 (공식적인 표준).

## 5. 임계구역 (Critical Section)

### 문제 상황

공유 자원에 여러 프로세스가 **동시에 접근**하면 데이터가 깨집니다.

> 🍹 **믹서기 비유**: 고기와 과일을 동시에 넣고 갈면 엉망이 됨 → 하나씩 따로 처리해야 함.

### 정의

**임계 구역**: 공유 자원에 접근하는 코드 영역. 동시에 **한 프로세스만** 들어가야 함.

> 🚻 **공중화장실 변기 비유**: 한 명만 들어가서 문 잠그고 볼 일 보고 나옴.

---

## 6. 임계구역 문제 해결의 3가지 조건

| 조건 | 의미 | 비유 |
|------|------|------|
| **1. 상호 배제 (Mutual Exclusion)** | 임계구역에 한 번에 **하나의 프로세스**만 진입 | 화장실 문 잠금 |
| **2. 한정 대기 (Bounded Waiting)** | 무한히 대기하지 않도록 보장 | 비어있는데 못 들어가면 X |
| **3. 진행의 융통성 (Progress Flexibility)** | 한 프로세스 문제로 **다른 프로세스 진행을 막지 않음** | 한 명이 멈췄다고 줄 전체가 멈추면 X |

### 기본 코드

```c
while (lock == true);  // 문 잠겼나 똑똑 (busy wait)
lock = true;           // 문 잠그기
// ── 임계구역 ──
lock = false;          // 문 열고 나옴
```

⚠️ 이 단순 코드는 **문맥 교환 타이밍에 따라 실패** (세 조건 모두 깨짐).

---

## 7. 임계구역 해결 방법

### 🔧 하드웨어 방법: Test-and-Set

CPU 명령어 수준에서 **원자적(Atomic)**으로 검사+지정을 수행.

```c
while (test_and_set(&lock) == true);
// 임계구역
lock = false;
```

단점: **바쁜 대기(Busy waiting)** 발생.

### 🧩 피터슨 알고리즘 (Peterson's Algorithm)

두 프로세스 간 임계구역 문제를 **소프트웨어**로 해결.

- `turn`, `flag[]` 변수 2개 사용
- 3가지 조건 모두 만족
- ⚠️ **2개 프로세스만** 가능 (한계)

### 🧩 데커 알고리즘 (Dekker's Algorithm)

2개 프로세스용. 피터슨과 유사하지만 복잡.

## 8. 세마포어 (Semaphore) ⭐

**임계구역 진입 전 스위치**. 공유되는 자원이 **N개**일 때 사용.

### 기본 구조

```c
Semaphore(RS = n);   // 공유 자원 n개 선언

P(RS) {              // 진입 시: 자원 1개 소모
    while (RS <= 0); // 자원 없으면 대기
    RS--;
}

// ── 임계구역 ──

V(RS) {              // 퇴출 시: 자원 1개 반납
    RS++;
    wake_up();       // 대기 중 프로세스에게 신호
}
```

### P / V 함수

| 함수 | 역할 |
|------|------|
| **P (Proberen)** | 감소 — **자원 획득** (락) |
| **V (Verhogen)** | 증가 — **자원 반납** (언락) |

### 동작 예시

```
초기:  Semaphore = 2 (자원 2개)
P1 P()  → RS=1  (P1 임계구역 진입)
P2 P()  → RS=0  (P2 임계구역 진입)
P3 P()  → RS=0이므로 대기 ⏳
P1 V()  → RS=1, P3 깨움 → RS=0 (P3 진입)
```

---

## 9. 뮤텍스 (Mutex) — 자원이 **1개일 때**

```
Mutex  = Mutual Exclusion
자원 1개 = 단일 락
```

- 세마포어의 **특수한 형태** (N=1)
- 단순하고 빠름

| 구분 | 뮤텍스 | 세마포어 |
|------|--------|----------|
| **자원 수** | 1개 | N개 |
| **용도** | 상호 배제 | 자원 계수 |

---

## 10. 모니터 (Monitor)

세마포어는 P/V를 직접 호출해야 해서 **잘못 사용**할 위험이 있습니다. 모니터는 이를 감싸 사용자에게 **인터페이스만** 제공합니다.

```
사용자 → increment() 호출
              ↓
       Monitor 내부에서
       자동으로 P() → 임계구역 → V()
```

- 사용자는 `increment()` 같은 함수만 호출
- **자바의 `synchronized`**가 대표적 예

## 📝 핵심 요약

1. 프로세스는 독립 메모리 공간 → **IPC 필요**
2. IPC 기법 7가지: **시그널, 인터럽트, 공유파일, 파이프, 메시지큐, 공유메모리, 소켓**
3. 인터럽트(이벤트 기반) ↔ 폴링(주기적 확인)
4. 소켓 = 택배 박스 (TCP/IP)
5. **임계구역 3조건**: 상호배제, 한정대기, 진행 융통성
6. 해결: **세마포어 (P/V)** ← 핵심! ⭐
7. **뮤텍스** = 자원 1개, **모니터** = 세마포어 감싼 인터페이스 (자바 `synchronized`)
