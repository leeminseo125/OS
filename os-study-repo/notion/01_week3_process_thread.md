## 1. 프로세스의 개념

### 🍳 요리 모형으로 보는 진화 과정

| 방식 | 특징 | 비유 |
|------|------|------|
| **일괄 작업 방식** | 큐에 담아 순서대로 한 번에 하나씩 처리 | 한 요리 끝날 때까지 대기 |
| **시분할 방식** | 여러 요리를 동시에 분할 처리 → 테이블 회전율↑ | 한 사람이 먹는 동안 다른 사람 요리 제공 |
| **시분할 예외 처리** | 재료가 없거나 준비가 안 된 경우, 보조에게 위임하거나 거부 | 새우 손질 → 보조에게 / 샐러드 재료 소진 → 장보기 or 거부 |

→ 이 예외 처리 패턴이 바로 **운영체제의 프로세스 상태 전이** 메커니즘과 매핑됩니다.

### 운영체제의 기능 (6가지)

프로세스 관리 · 메모리 관리 · 파일시스템 관리 · 입출력 장치 관리 · 네트워크 관리 · **IPC(프로세스 간 통신)**

> 💡 **중간고사 범위**: 프로세스 관리 + IPC

### 프로그램 vs 프로세스 vs 프로세서 — 헷갈리지 말 것!

| 용어 | 정의 | 위치 |
|------|------|------|
| **프로그램(Program)** | 하드디스크에 저장된 소프트웨어 | HDD |
| **프로세스(Process)** | 메모리에 적재되어 실행 중인 프로그램 | Memory |
| **프로세서(Processor)** | 명령어를 실행하는 하드웨어 = **CPU** | Hardware |

### 프로그램 → 프로세스 전환

```
프로그램 + PCB + 사용자 데이터 = 프로세스
프로세스 - PCB - 사용자 데이터 = 프로그램
```

프로그램이 메모리에 올라올 때 **순번표(PCB)**를 커널 영역에 제출하고, 사용자 영역에 자리 잡습니다. 종료 시 PCB와 함께 메모리에서 제거됩니다.

> 🍴 식당 비유: 순번표 받고 → 제출하고 → 테이블 배정 → 식사 → 퇴장

## 2. 프로세스 제어 블록 (PCB)

**정의**: 운영체제가 관리하는 자료구조로, 모든 프로세스에 **1:1로 대응**됩니다.

### PCB 구성 요소

| 필드 | 설명 |
|------|------|
| **포인터(Pointer)** | PCB를 Linked List로 연결하기 위한 포인터 |
| **프로세스 상태** | 생성/준비/실행/대기/보류/종료 |
| **PID(Process ID)** | 프로세스 고유 식별 번호 (중복 불가) |
| **프로그램 카운터(PC)** | 다음 실행할 명령어 주소 (ISR 복귀용) |
| **우선순위, 레지스터 정보** | 스케줄링 및 문맥교환용 |
| **메모리 관리 정보** | 베이스 레지스터, 한계 레지스터 등 |
| **할당된 자원 정보** | 파일, I/O 장치 등 |
| **계정 정보** | CPU 사용 시간, 실행 횟수 |
| **PPID & CPID** | 부모/자식 프로세스 ID (계층구조용) |

> 🔍 **눈으로 확인하기**: `Ctrl + Shift + ESC` → 작업 관리자에서 PID, 상태 등을 볼 수 있음. 이게 바로 PCB의 일부 내용!

### 인터럽트 루틴과 PC

```
일반 실행 → 외부 이벤트 발생 (인터럽트)
          ↓
     PC에 다음 주소 저장
          ↓
   ISR (Interrupt Service Routine) 실행
          ↓
      PC 기준으로 복귀 후 계속 실행
```

## 3. 프로세스의 상태 (State Diagram)

### 기본 상태

```
      ┌────────┐
      │  생성  │ (New)
      └────┬───┘
           ↓
      ┌────────┐   dispatch   ┌────────┐
      │  준비  │ ───────────→ │  실행  │
      │ Ready  │ ←──────────  │Running │
      └────────┘   timeout    └───┬────┘
           ↑                      ↓
           │                  ┌───────┐
           └──────── wake ────│ 대기  │
                              │ Block │
                              └───────┘
                                  ↓
                              ┌───────┐
                              │ 종료  │
                              └───────┘
```

### 상태 상세

| 상태 | 의미 |
|------|------|
| **생성(New)** | PCB 생성, 메모리 적재 준비 |
| **준비(Ready)** | 준비 큐에서 CPU 할당 대기 |
| **실행(Running)** | CPU를 점유하고 코드 실행 중 |
| **대기(Block/Wait)** | I/O 완료 등을 기다림 |
| **보류(Suspend)** | 메모리 밖(스왑 영역)으로 밀려난 상태 |
| **종료(Terminate)** | 프로세스 소멸, PCB 삭제 |

### 중요 전이

- `준비 → 실행`: **dispatch**
- `실행 → 준비`: **timeout** (시분할!)
- `실행 → 대기`: I/O 요청 등
- `대기 → 준비`: I/O 완료 → 다시 줄 서기
- `대기 → 보류대기`: 너무 오래 기다리면 메모리에서 퇴출
- `보류대기/보류준비`: 메모리 밖(스왑) 영역에 존재

## 4. 문맥 교환 (Context Switching)

**정의**: CPU를 점유한 프로세스가 바뀔 때, PCB 정보를 갱신하고 다른 프로세스 상태를 복원하는 과정.

```
Process A 실행
    ↓
PCB A 저장 (현재 상태 → PCB A)
    ↓
PCB B 복원 (PCB B → 레지스터)
    ↓
Process B 실행
```

### 타임 슬라이스(Time Slice)

| 길이 | 영향 |
|------|------|
| **너무 김** | 일괄 처리와 유사, 효율 ↓ |
| **너무 짧음** | 문맥교환 오버헤드 ↑ → 오히려 성능 ↓ |
| **적정 (10~20ms)** | 운영체제가 결정 |

## 5. 프로세스의 구조 (메모리 레이아웃)

식당 테이블 위 의자 4개처럼, 프로세스는 메모리에 **4개 영역**을 가집니다.

```
┌─────────────────┐  ← High address
│  스택(Stack)    │  ↓ 아래로 자라남 (동적)
├─────────────────┤     - 지역변수, 매개변수, 함수 호출 스택
│        ↕        │
│                 │  ← 충돌 가능 (Stack Overflow / Heap Overflow)
│        ↕        │
├─────────────────┤
│   힙(Heap)      │  ↑ 위로 자라남 (동적)
│                 │     - malloc, new 등 동적 할당
├─────────────────┤
│  데이터(Data)    │  (정적) — 전역/static 변수
├─────────────────┤
│  코드(Code/Text)│  (정적) — 프로그램 코드
└─────────────────┘  ← Low address
```

| 영역 | 할당 시점 | 크기 결정 |
|------|----------|----------|
| 코드 (Code) | 정적 | 컴파일 타임 |
| 데이터 (Data) | 정적 | 컴파일 타임 |
| 힙 (Heap) | 동적 | 런타임 |
| 스택 (Stack) | 동적 | 컴파일 타임 (고정) |

## 6. 프로세스의 생성과 복사 — fork()

**fork()**: 현재 프로세스를 복사하여 **자식 프로세스**를 생성하는 시스템 호출.

### fork() 시 PCB 변화

| 필드 | 부모 (A) | 자식 (B) |
|------|----------|----------|
| PID | 100 | **101** (새 번호) |
| PPID | 10 | **100** (부모 PID) |
| CPID | 101 | 0 (자식 없음) |
| 코드 영역 주소 | 0xA000 | **0xA000** (공유) |
| 데이터 영역 주소 | 0xB000 | **0xB100** (분리) |

### 왜 fork()를 쓰는가?

- **속도가 빠르다** — 기존 자원을 최대한 재활용
- **계층 구조 관리** — 부모-자식 트리 형태로 효율적
- **자원 회수 용이** — 부모가 자식 자원 회수 책임

---

## 7. 프로세스의 전환 — exec()

**exec()**: 기존 프로세스의 메모리 영역을 덮어씌우고 **새로운 프로그램으로 교체**하는 시스템 호출.

```
로그인 프로세스 (fork하지 않고)
     ↓ exec()
 shell 프로세스  ← 같은 메모리 영역을 재사용
```

- PID, PPID는 그대로 유지
- 코드/데이터 영역만 새 프로그램으로 교체
- 메모리 효율 ↑

## 8. 프로세스의 계층 구조 (Unix)

```
            init (PID 1)
          /   |    \   \
      login login login login
        |     |     |    |
      bash  bash  bash  bash
       |     |     |     |
      app1  app2  app3  app4
```

### 장점

- **체계적 관리**: 트리 형태로 전체 프로세스 추적
- **효율적 생성**: fork로 빠른 복제
- **자원 회수 용이**: 부모가 자식 자원 회수 책임

### 예외 상황

#### 🧟 고아 프로세스 (Orphan Process)

```
부모가 먼저 죽음
    ↓
자식만 남음 (계층에서 떨어져 나옴)
    ↓
init 프로세스가 입양 → 관리 재개
```

> 자바의 가비지 컬렉션과 유사한 매커니즘

#### 🧟‍♂️ 좀비 프로세스 (Zombie Process)

```
자식이 먼저 죽음 ("나 끝났어요!")
    ↓
부모가 신호를 못 들음 (회수 X)
    ↓
자식 PCB만 메모리에 잔존 → 자원 낭비
```

> **문제**: 좀비가 누적되면 새 프로세스 생성 불가

## 9. 스레드 (Thread)

**정의**: 프로세스 내부에서 실행되는 **더 작은 실행 단위**.

```
┌────────────────────────────────┐
│     프로세스 (한글 앱)          │
│  ┌─────┐ ┌─────┐ ┌─────┐      │
│  │ 편집 │ │ 맞춤법 │ │저장 │    │  ← 각각 스레드
│  │Thread│ │Thread│ │Thread│    │
│  └─────┘ └─────┘ └─────┘      │
│   Code | Data | Heap | Stack   │  ← 공유
└────────────────────────────────┘
```

- 한 프로세스 내 스레드들은 **Code, Data, Heap 영역 공유**
- **Stack은 스레드별로 분리**

### TCB (Thread Control Block)

프로세스의 PCB처럼, 스레드도 **TCB**라는 제어 블록을 가집니다.

## 10. 멀티 스레드 / 태스킹 / 프로세싱

| 용어 | 의미 | 예 |
|------|------|-----|
| **멀티 스레드** | 한 프로세스 내 여러 스레드 동시 실행 | 한글 앱 |
| **멀티 태스킹** | 여러 프로세스(태스크) 동시 실행 (IPC 필요) | 여러 앱 동시 사용 |
| **멀티 프로세싱** | **여러 CPU**에서 여러 프로세스 동시 처리 | 다중 코어 |
| **하이퍼스레딩** | 하나의 CPU가 2~3개 가상 CPU처럼 동작 | Intel HT 기술 |

### 멀티스레드의 장단점

**장점** ✅
- 응답성 향상 — 한 스레드가 막혀도 다른 스레드가 응답
- 자원 공유 — Code/Data/Heap 공유로 효율적
- 효율성 — 적은 자원으로 여러 작업

**단점** ❌
- **동기화 필요** — 공유 자원 충돌
- **한 스레드가 죽으면 전체 프로세스가 영향받음**

> 🌐 **IE vs Chrome**: IE는 멀티스레드(탭 하나 죽으면 다 죽음), Chrome은 멀티프로세스(탭 독립)

## 11. 멀티 스레드 모델 (User Thread ↔ Kernel Thread 매핑)

사용자 레벨 스레드를 커널 레벨 스레드에 어떻게 매핑할지 결정.

| 모델 | 매핑 | 특징 |
|------|------|------|
| **다대일 (N:1)** | N개 User → 1개 Kernel | 일괄 처리처럼 순차 실행, 한 스레드 블로킹 시 전체 대기 |
| **일대일 (1:1)** | 1개 User → 1개 Kernel | 병렬 처리 가능, 빠름 (Windows, Linux) |
| **다대다 (N:M)** | N개 User → M개 Kernel | 유연성과 효율의 균형 |

---

## 📝 핵심 요약

1. **프로그램**(HDD) → **프로세스**(Memory, PCB 포함) → **프로세서**(CPU)
2. **PCB = 프로세스의 순번표**, 커널 영역에 저장
3. **문맥 교환** = PCB 저장/복원, 10~20ms가 적정
4. **fork()** = 복사, **exec()** = 이미지 교체
5. **고아 프로세스** = 부모가 먼저 죽음 (init이 입양), **좀비** = 자식 회수 실패
6. **스레드**는 프로세스 내부 실행 단위, Code/Data/Heap 공유
7. **멀티스레드 모델**: N:1, 1:1, N:M
